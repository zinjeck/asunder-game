6. CONFIRMED TASK AND ATTENDANCE FOUNDATION
===============================================================================

The current task record was intentionally minimal for the first Work behavior.
It includes concepts equivalent to:

- kind
- source
- phase
- priority
- target_object_id
- start_world_minute
- player_locked

Current kinds include None and Work. Current phases include None, Pending,
Traveling, Performing, and Blocked.

Active tasks are tracked through a sorted active-task ID registry and lookup.
Task changes have a focused version counter. Existing saves/citizens are
migrated through the task-state ensure path.

Current attendance is derived rather than stored as a second truth. A citizen
attends a workplace only when all relevant facts agree:

- the citizen is alive;
- the citizen is assigned to that workplace;
- current task kind is Work;
- current task target matches the workplace;
- current task phase is Performing;
- movement is Idle;
- the authoritative citizen tile is a currently approved attendance tile.

Do not add a redundant persistent attendance boolean.

Current limitation: approved attendance tiles still effectively use exterior
workplace access tiles. The new activity-location policy frontier must replace
that hardcoded assumption.

7. NAVIGATION PERFORMANCE FIX — CONFIRMED WORKING
===============================================================================

The user reported two linked failures:

- Work became Blocked despite a valid long route.
- Pressing P produced a lag spike that grew with distance.

The diagnostic screenshot showed:

    Path status: success
    Path cost: 401
    Expanded nodes: 18,941
    Search cost: approximately 609 ms
    Historical maximum simulation tick: approximately 1,211 ms

The cause was confirmed:

- autonomous Work allowed 10,000 expanded nodes;
- debug P allowed 50,000;
- exact A* expanded 18,941 nodes to prove the shortest route;
- Work hit its cap and became Blocked while P eventually succeeded;
- multiple workplace path requests could stack in one simulation tick.

Latest applied and runtime-confirmed direction:

- CityNavigationSystem default maximum expanded nodes: 10,000.
- Deterministic weighted heuristic: HEURISTIC_WEIGHT = 2.
- Routes remain valid and deterministic but may be slightly longer than the
  mathematical optimum.
- Single-destination Manhattan calculation has a fast path.
- CitizenTaskSystem admits one new workplace path request per tick.
- Empty-tile citizen walkability avoids allocating an empty Dictionary and
  performing an object lookup for every A* neighbor.

Do not casually restore exact shortest-path proof or a 50,000-node synchronous P
request. That would reintroduce visible main-thread stalls. If exact optimal
paths become important later, use a bounded time-sliced or hierarchical system.

8. USER’S SETTLED HOMELESSNESS AND INVENTORY INTENT
===============================================================================

A citizen without a valid assigned house is homeless. Do not assign the City
Keep as a fake home merely to reuse Home behavior.

Homeless citizen rules:

- They still complete Work and other real tasks normally.
- When genuinely free, they should wander locally around the City Keep.
- They should not receive a Return Home task.
- They should not care about or receive access to private-home inventory.
- Their personal inventory remains relevant because it travels with them.
- Wandering must never override Work, a player-locked task, or a future critical
  need.

Housed citizen rules:

- A citizen may return only to the home identified by its valid home_object_id.
- When going home or remaining at home, the citizen should occupy the building’s
  interior/footprint tiles rather than waiting outside on an access tile.
- Private-home inventory access must eventually require a real home relationship
  and the appropriate access/location rules.

Do not globally mark House footprints walkable. Doing so would allow strangers
to enter every house and would let routes use buildings as shortcuts.

9. DATA-DRIVEN ACTIVITY-LOCATION PRINCIPLE
===============================================================================

The user explicitly wants a universal template configured by aspects.

Good architecture:

    generic resolver reads a location/movement policy
        -> building definition selects policy modes and parameters
        -> task chooses a concrete target tile
        -> navigation receives contextual traversal permission
        -> movement follows the route
        -> attendance checks the same resolved policy

Bad architecture:

    if object_type == FISHING_GROUNDS:
        special Fishery movement
    elif object_type == BAKERY:
        special Bakery movement

It is acceptable and necessary to implement generic policy modes in a match.
It is not acceptable to branch on each concrete building identity when a mode
can represent the behavior.

The current WorldData definition schema already contains useful scaffolding:

- work_location_policy
- work_movement_policy
- break_location_policy
- resource_source_policy
- overflow_policy

Existing work-location modes include concepts such as resource-source tiles,
linked tiles, workstations, explicit points, and footprint.

Existing work-movement modes include move-between-work-points, station-based,
remain-at-station, and linked-tile tasks.

The Fishing Grounds definition currently selects a resource-source-oriented
location mode and move-between-work-points, but CitizenTaskSystem does not yet
consume these policies. The next pass should activate a generic policy layer,
not add Fishery-specific task code.

10. FISHING GROUNDS WORK BEHAVIOR — USER REQUIREMENT
===============================================================================

Fishing workers should visually simulate fishing along the coast inside the
workplace’s collection zone.

Candidate standing tile rules:

- The tile must be inside the Fishing Grounds collection zone.
- The standing tile must be walkable land.
- The standing tile must cardinally border water.
- The rule must not depend on whether the neighboring water contains the Fish
  resource.
- Environmental fish resources still determine site productivity separately.
  Work-position selection and resource productivity are different concerns.

Behavior while working:

1. Choose one valid water-border land tile.
2. Travel there.
3. Remain there for a configured amount of simulation time.
4. Choose another valid water-border land tile.
5. Travel to the new point and repeat while the shift remains active.

Anti-oscillation requirements:

- Do not choose the current tile when alternatives exist.
- Exclude at least the immediately previous work point when alternatives exist.
- Prefer a configurable minimum Manhattan distance so movement is visible.
- Avoid repeatedly swinging between two tiles.
- Stay put only when no valid alternative exists.

“Random” choices must be deterministic simulation choices. Use controlled data
such as citizen ID, workplace ID, relocation count, and/or simulation tick. Do
not use frame timing or uncontrolled global randomness.

This should be represented through generic modes, for example conceptually:

    location mode: zone tile adjacent to configured terrain
    zone source: workplace collection/resource zone
    stand terrain: land
    adjacent terrain: water

    movement mode: relocate after dwell
    dwell duration: configured range
    minimum relocation distance: configured
    recently avoided points: configured count

Names and exact schema must be chosen after inspecting the newest code. Do not
freeze these conceptual names without that inspection.

11. NON-EXTRACTION AND INDOOR WORKPLACES
===============================================================================

Not all jobs extract environmental resources. The generic system must support
indoor and station-based work without resource-zone assumptions.

Example future Bakery:

- Work location may be an interior footprint tile or explicit workstation.
- Movement policy may be remain-at-station.
- Workers should not roam a resource collection zone.

Other reusable examples:

- gathering workplace: zone-adjacent work points + relocation;
- workshop: explicit workstations + station-based movement;
- large interior workplace: footprint/interior points + optional relocation;
- linked-field workplace: linked tiles + linked-tile tasks;
- guard post: explicit points + remain-at-station or patrol sequence.

The task system should ask the workplace policy for valid locations. It should
not infer behavior from output_resource or workplace_kind alone.

12. CONTEXTUAL BUILDING INTERIORS
===============================================================================

Current global citizen walkability treats ordinary building footprints as
blocked. Home interiors and future indoor workplaces require contextual access.

Required principle:

- A tile may be globally blocked but permitted for a specific task/actor/target
  context.
- A resident may enter the footprint of their own assigned home.
- A worker may enter the target workplace only when its location policy allows
  interior work.
- Unrelated buildings remain blocked.
- Manual/debug navigation without such permission should not route through
  buildings.
- Future guests require explicit permission rather than universal access.

Navigation and movement must use the same contextual rule. It is not sufficient
for pathfinding to create an interior route if CitizenMovementSystem later
rejects the same footprint tile as blocked.

Do not make all building footprints globally walkable.

17. INVARIANTS TO PRESERVE
===============================================================================

- Movement state answers how the citizen moves.
- Task state answers why the citizen moves and what happens on arrival.
- Work/location policy answers where and how that type of activity may occur.
- Environmental resource policy answers how the site gains productivity.
- These concerns must remain separate.
- Citizen authoritative positions and spatial buckets must agree.
- Active mover and active task arrays/lookups must agree.
- Task target tiles must be valid or use an explicit invalid sentinel.
- A contextual interior path must be legal under both route search and movement
  advancement.
- Dead, unassigned, absent, traveling, or blocked workers do not contribute as
  productive attendees.
- Resource overflow is never silently deleted.
- Ground resources are not city storage.
- Player-locked tasks are never silently replaced by idle wandering.
