2. IMPORTANT RECONCILIATION RESULTS

Aggregate city resource accounting

    Older rule:
        The top resource bar displays public Stockpiles only, and fish still
        inside a workplace is excluded.

    Newer accepted rule:
        Citywide stored totals include resources physically stored in eligible
        building containers, including workplaces and future real house
        storage. Open-tile ground piles remain excluded. Public Stockpile totals
        remain separately queryable.

    Code evidence:
        The inspected top bar calls the aggregate stored-amount and aggregate
        capacity queries. The eligibility helper excludes container type none
        and ground_pile. Fishing Grounds storage therefore contributes.

    Documentation treatment:
        New rule is current. Old Stockpile-only bar is preserved only as a
        superseded decision. Public storage remains a valid separate category.

Workers eating during work

    Earlier shorthand:
        Workers do not eat while actively working; they eat during scheduled
        workplace breaks.

    Later refinement:
        Ordinary work still does not include arbitrary eating. Scheduled breaks
        remain workplace-local eating/social windows, but severe graded hunger
        may suspend a task, resolve food, and later resume the task.

    Documentation treatment:
        The later nuanced rule is current. Scheduled breaks and emergency need
        interruption are separate mechanisms. Exact hunger values mentioned in
        conversation remain proposed tuning anchors, not hard rules.

Production destination

    Earlier concept:
        Working citizens might fill personal inventories with produced fish
        before excess enters workplace storage.

    Current architecture and handoff:
        Clock-driven production writes output to workplace-local storage first.
        Citizen inventory is for later carrying and eating.

    Documentation treatment:
        Workplace-first output is current. The personal-inventory-first concept
        is listed as superseded.

Clock ownership

    Earlier plan:
        A thin clock might be advanced by the city renderer.

    Current accepted architecture:
        SimulationClock owns time, SimulationCoordinator owns ordered dispatch,
        WorldData exposes the authoritative entry, and renderers observe.

    Code evidence:
        Both clock and coordinator are autoloads. The coordinator receives the
        tick signal and calls the ordered simulation entry. No renderer advances
        the clock.

    Documentation treatment:
        Global clock/coordinator path is current; renderer-owned time is
        superseded.

Region water rule

    Older rule:
        A fixed small count of water/river tiles invalidated the 9x9 region.

    Current rule:
        Only more than 90 percent ocean invalidates. Rivers are exempt.

    Code evidence:
        Region validation counts only ocean biome. The selected outline is cyan.
        A stale console message still mentions ocean/river.

    Documentation treatment:
        Ocean-only rule is current. Stale wording is recorded as a minor issue.

River generation

    Current code:
        A working heuristic includes source qualification, a simple valley-like
        source test, mountain penalties, meander noise, and open-ocean routing.

    Deferred design:
        Fully intelligent valley/catchment-aware river generation was explicitly
        postponed.

    Documentation treatment:
        Current rivers are a working prototype. The presence of a valley-source
        heuristic is not treated as completion of the advanced deferred design.

Godot and C++

    Explored direction:
        Leave Godot early and port before the project becomes larger.

    Settled current direction:
        Continue Godot/GDScript, keep simulation data-oriented, and consider
        GDExtension only for stable measured hot loops. Do not perform a
        wholesale C++ or premature ECS rewrite.

    Documentation treatment:
        Godot/GDScript is current. Future native code is conditional, not a
        commitment or rejection forever.


3. IMPORTANT UNCERTAINTIES

Snapshot date

    The archive identifier is recorded, but the commit date is unknown. ZIP
    entry timestamps are not treated as Git evidence.

Runtime status

    The user and handoff confirmed the current checkpoint, but this audit did
    not run Godot. Parsing, visual alignment, exact timings, and interactive
    behavior still require a local test.

Citizen inventory and aggregate totals

    Current aggregate queries scan city objects only. Personal inventories are
    physical containers but are not included. The latest explicit accounting
    rule named buildings, houses, workplaces, and ground exclusion, not carried
    inventory. Because citizen inventories remain empty in normal play, this has
    not yet surfaced. The docs mark the choice as Needs Verification rather than
    inventing an answer.

House storage

    Houses are classified as private_home_storage, but their current definition
    allows no resources and gives no capacity. The design says legitimate house
    storage should contribute once real storage exists; the current house
    implementation contributes zero.

City versus empire architecture

    The intent for cities to participate in empires is clear. Exact ownership,
    off-screen simulation, political boundaries, multi-city loading, trade, and
    strategic aggregation contracts are not yet defined.

Persistence timing

    Real disk save/load is absent. A later persistence phase is sensible and
    consistent with project direction, but no exact milestone, format, or schema
    strategy has been accepted. It is recorded as proposed/deferred.

Hunger tuning

    Qualitative behavior is settled; the example values around 70-100, 50, and
    30 are not fixed thresholds. Drain rates, task scores, starvation effects,
    and emergency exceptions require future runtime tuning.

Food-source search details

    Personal, home, and Stockpile priority has been discussed, but exact access
    radius, travel choice, emergency borrowing, ownership, and failure behavior
    remain future work.

Road authority boundary

    Normal UI placement prevalidates roads, but the low-level road-add operation
    does not repeat every terrain check. This is documented as a state-boundary
    risk, not asserted as a current player-facing bug.

Unused and temporary symbols

    CITY_OBJECT_PLACEHOLDER_BUILDING, a legacy city resource dictionary, old
    hover-panel helpers, and temporary scene files appear unused. They should be
    verified before removal; no deletion was made during this documentation
    task.


4. DETAILS INTENTIONALLY EXCLUDED OR DE-EMPHASIZED

The three repository documents intentionally avoid becoming a second codebase.
The following were generally excluded unless needed to explain a current
checkpoint or test:

    line numbers
    exact function bodies
    temporary local variable names
    exact node paths beyond current scene/script navigation
    UID values
    every signal connection
    every helper method
    citizen name-pool contents
    exact UI offsets, sizes, margins, fonts, and border widths
    exact RGBA values except stable qualitative color conventions
    every noise frequency, octave, threshold, and resource spawn chance
    texture warmup row counts
    exact camera speeds and zoom limits
    exact current production arithmetic constants
    implementation-specific dictionary-copy mechanics
    exhaustive print/debug messages
    import metadata and icon details

Some current values appear in CURRENT_STATE.md because they materially orient
the current prototype, such as the starting population, current map dimensions,
current building capacities, and the 9x9 region rule. They are explicitly
presented as current checkpoint values rather than permanent architecture.

Exact symbols are used selectively as repository search anchors, especially for
autoloads, central state, renderer/controller scripts, validator, production
system, shared visuals, cache, and debug panel. The documents repeatedly state
that paths and symbols may move while responsibilities remain.


5. FINAL CONSISTENCY AUDIT

The following cross-document checks were performed:

    AGENTS.md describes durable rules and does not carry the current task status.

    CURRENT_STATE.md identifies Phase 2C production validation as the immediate
    frontier and does not claim source providers, ground piles, schedules,
    hunger behavior, hauling, movement, or save/load are implemented.

    DECISION_LOG.md records aggregate city accounting as the replacement for
    the older Stockpile-only top bar.

    All three documents distinguish aggregate accounting from physical access.

    All three exclude ground piles from stored-city totals.

    All three preserve workplace-local output as the first production
    destination.

    All three describe one global clock/coordinator path and prohibit renderer-
    driven simulation.

    All three describe citizens as data records first and visible actors later.

    All three distinguish employment, attendance, active work, breaks, tasks,
    and off-duty availability.

    All three preserve scheduled workplace-local breaks while allowing future
    urgent hunger interruption.

    All three label exact hunger thresholds as tunable.

    All three treat advanced valley-aware rivers, visible citizen animation,
    wholesale C++ migration, and disk persistence as deferred rather than
    current implementation.

    Current representative files are provided as navigation anchors, not fixed
    architecture contracts.

    No document claims that this documentation replaces current code inspection
    or runtime testing.


6. RECOMMENDATIONS FOR KEEPING THE DOCUMENTATION CURRENT

AGENTS.md

    Change only when a durable source-of-truth rule, state-ownership principle,
    project-wide safety rule, or long-term architecture boundary changes.

    Do not update it for:
        function renames
        file moves
        tuning values
        current bugs
        phase progress
        isolated UI changes

CURRENT_STATE.md

    Update after:
        completing Phase 2C validation
        completing a roadmap phase
        changing the immediate frontier
        a major refactor that moves ownership
        introducing real persistence
        confirming a significant regression or fix

    Preserve:
        snapshot/commit identifier
        revision date
        runtime verification status
        system maturity labels
        immediate tests and risks

DECISION_LOG.md

    Add an entry when:
        gameplay meaning changes
        accounting/access rules change
        an ownership boundary moves
        simulation order changes
        an old design is rejected or superseded
        engine/performance strategy changes
        a phase contract becomes settled

    Do not add an entry for:
        trivial renames
        one-off syntax fixes
        temporary colors or dimensions
        every completed task
        speculative ideas that were never accepted

Repository practice

    Commit the three documentation files with a message that identifies the
    checkpoint. Keep documentation changes visible in review rather than hiding
    them inside unrelated code commits.

    When a decision changes, update the old entry's status or add a replacement
    and the superseded index. Do not erase the old rule if doing so could cause
    it to return later.

    When a responsibility moves files, update only the file-orientation tables
    and any misleading current-location notes. Do not rewrite the conceptual
    architecture section if the ownership contract did not change.

    When a new phase begins, move the prior frontier into completed/current
    status, record what was actually verified, and make the next dependency gate
    explicit.

    Before any whole-file replacement, obtain the newest file or repository
    state and compare it with confirmed post-snapshot edits.

    Add deterministic automated simulation tests as soon as the Phase 2C/2D
    contracts stabilize. Until then, keep the manual verification checklist
    current and record user-confirmed results.
